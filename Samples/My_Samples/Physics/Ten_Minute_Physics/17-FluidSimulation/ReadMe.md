## Recommended Reading

Although the code is commented to be as self-explanatory as possible, it is strongly recommended to first read the first 5 chapters of the [SIGGRAPH 2007 fluid simulation notes by Bridson and Müller-Fischer](https://www.cs.ubc.ca/~rbridson/fluidsimulation/). They are only about the first 50 pages of the course notes and should not be particularly difficult to follow if you already have basic graphics / numerical-methods background.

Those chapters cover the exact concepts this sample relies on most heavily: Eulerian grids, advection, incompressibility, pressure projection, and smoke-oriented extensions such as vorticity confinement.

## Overview

This sample demonstrates real-time 3D smoke simulation on a staggered MAC grid, with all simulation steps executed on the GPU through compute shaders and the result rendered with a separate volumetric ray-marching pass.

The sample includes:

- A draggable solid obstacle sphere that injects motion into the surrounding fluid
- A draggable smoke source sphere used both for smoke injection and for a faint visual indicator in the volume render
- Multiple visualization modes: smoke, pressure, and velocity
- A fully GPU-resident pipeline with 3D textures, ping-pong buffers, and compute-based rendering

## Key Features

- **Eulerian 3D Fluid Solver** - Simulation runs on a fixed voxel grid instead of tracking particles
- **Staggered MAC Layout** - Velocity components are stored on cell faces, while pressure / smoke / solid flags are cell-centered
- **Pressure Projection** - Velocity is made approximately divergence-free through a pressure solve followed by gradient subtraction
- **Interactive Obstacle Coupling** - Dragging the obstacle updates both position and obstacle velocity, which is injected into the solver
- **Volumetric Rendering** - The fluid is visualized with a compute shader that ray-marches the 3D simulation textures into a 2D overlay

## Simulation Pipeline

Each frame executes the following GPU simulation steps:

1. **Set Obstacle** - Marks domain walls and interior obstacle cells in the solid field. For obstacle cells, the shader also writes the obstacle velocity into the face-centered velocity field so a moving sphere can push the smoke instead of behaving like a static blocker.
2. **Apply Forces / Sources** - Adds external effects before incompressibility is enforced. In this sample that means buoyancy, optional vorticity-related terms, and smoke injection around the source sphere so later steps advect something meaningful.
3. **Boundary Pass** - Reapplies wall and obstacle boundary conditions after forces. This is important because the previous step may have introduced face velocities that are incompatible with solid cells or no-penetration constraints.
4. **Compute Divergence** - Measures how much net flow enters or leaves each fluid cell. This is the quantity the pressure solve tries to cancel, because incompressible flow should have near-zero divergence.
5. **Pressure Solve** - Runs iterative Red-Black Gauss-Seidel with over-relaxation (SOR). This does not directly update velocity yet; it computes the scalar pressure field required to compensate for the divergence generated by the previous steps.
6. **Projection** - Subtracts the discrete pressure gradient from the face velocities. This is the step that turns the provisional velocity into an approximately divergence-free field.
7. **Boundary Pass Again** - Re-enforces obstacle and wall conditions after projection. This is necessary because projection modifies velocities near solids and can slightly violate the boundary constraints that were valid before pressure correction.
8. **Advect Velocity** - Backtraces through the current velocity field and resamples velocity with semi-Lagrangian advection. This transports momentum through the domain and is unconditionally stable, though numerically dissipative.
9. **Advect Smoke** - Uses the same backtracing idea for the scalar smoke density field. This is why the smoke visually follows the flow produced by the velocity solve.
10. **Optional MacCormack Correction** - When enabled, applies a higher-order correction to reduce the excess blurring introduced by first-order semi-Lagrangian transport. This is mainly a visual quality improvement layered on top of the base advection step.
11. **Swap / Ping-Pong** - Exchanges current and temporary textures so the next frame reads the newly computed state instead of the previous one.

In short, the frame alternates between:

- building a valid velocity field under forces and constraints
- projecting it back to incompressibility
- transporting both velocity and smoke forward in time

## Why The Steps Are Ordered This Way

- Forces are applied before projection because they create a provisional velocity that must then be corrected to satisfy incompressibility.
- Boundary conditions are enforced more than once because several later passes can disturb valid wall / obstacle velocities.
- Advection happens after projection so the transported velocity already reflects the incompressible field for the current step.
- Smoke is advected after velocity because the smoke field is passive: it follows the flow but does not define it.

This ordering is the standard split-step logic behind many practical Eulerian smoke solvers.

## Rendering Pipeline

Rendering is intentionally separate from the simulation update:

1. **Regular Scene Rendering** - Wicked Engine renders the ground plane and debug helper primitives, such as the domain wireframe box and the emitter wireframe sphere drawn from `sample.cpp`.
2. **Volume Ray Marching** - `FluidSimulation3D::RenderVolume()` dispatches `fluid_raymarching.hlsl`, which reconstructs one camera ray per pixel, intersects it with the simulation domain, marches through the 3D textures, and accumulates color / opacity front-to-back.
3. **Special Region Shading** - During the ray march, the obstacle sphere is treated as an opaque shaded sphere, while the smoke source sphere contributes only a small warm volumetric glow so it remains visible without looking like a solid mesh.
4. **2D Compositing** - The ray-marching pass writes a screen-sized RGBA texture. `SampleRenderPath::Compose()` then composites that texture over the already rendered 3D scene using premultiplied alpha.
5. **GUI Rendering** - The interface is drawn last so it stays above both the scene and the fluid overlay.

This approach keeps the rendering path simple and easy to reason about: the fluid is rendered as a volumetric overlay, not as standard triangle geometry.

## Technical Choices

| Aspect | Choice | Why |
|--------|--------|-----|
| Grid | 3D staggered MAC grid | Avoids odd-even pressure artifacts and gives a natural discrete divergence / gradient pair |
| Solver | Red-Black Gauss-Seidel + SOR | Parallel-friendly and converges faster than plain Jacobi for this use case |
| Advection | Semi-Lagrangian | Stable for interactive real-time use |
| Storage | 3D float textures | Convenient GPU storage with filtered sampling for advection / rendering |
| Rendering | Compute ray marching | Good fit for smoke volumes that do not have an explicit triangle surface |

- **Right Mouse / Middle Mouse** - Rotate camera

- **GUI Buttons** - Run / Pause simulation, Reset
